## Slide 6: เตรียม Enum ก่อนสร้าง Table

### File

```text
Supabase SQL Editor
```

### ทำไมยังไม่สร้าง table ทันที

เพราะ table ใน Slide 7 จะมี column นี้:




Enum ช่วยจำกัดค่าที่ database รับได้ เหมือน union type ใน TypeScript

## Slide 7: สร้าง Table `issues`

### File

```text
Supabase SQL Editor
```

### ตำแหน่งที่รัน

รัน SQL นี้ต่อจาก enum `issue_status` ที่สร้างไว้ก่อนหน้าใน Supabase SQL Editor

### ทำไมต้องเขียน `public.issues`

ใน Supabase table จะอยู่ใน schema ของ Postgres

- `public` คือ schema เริ่มต้นที่ app ใช้งานผ่าน Data API ได้
- `issues` คือชื่อ table ที่เราสร้าง
- เขียนเต็มเป็น `public.issues` เพื่อบอกชัด ๆ ว่า table นี้อยู่ใน schema ไหน

หลังจากนี้เวลาเขียน policy, insert หรือ select เราจะอ้างถึง table เดิมด้วยชื่อ `public.issues`

```sql
create table public.issues (
  id uuid primary key default gen_random_uuid(),
  reporter_name text not null,
  reporter_email text not null,
  title text not null check (char_length(title) >= 5),
  description text not null check (char_length(description) >= 10),
  status issue_status not null default 'OPEN',
  admin_comment text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

### อ่านคำสั่ง SQL ชุดนี้

- `uuid`: เก็บ id เป็นรหัสยาวแบบ unique
- `primary key`: ใช้ column นี้ระบุ row หลักของ table
- `default gen_random_uuid()`: ถ้าไม่ส่ง id มา database จะสร้าง id ให้อัตโนมัติ
- `text`: เก็บข้อความ
- `not null`: ห้ามเว้นว่าง
- `check (char_length(title) >= 5)`: title ต้องยาวอย่างน้อย 5 ตัวอักษร
- `status issue_status`: status ต้องเป็นค่าที่ enum `issue_status` อนุญาต
- `timestamptz`: เก็บวันเวลาแบบมี timezone เหมาะกับ `created_at` และ `updated_at`
- `default now()`: ถ้าไม่ส่งเวลามา database จะใส่เวลาปัจจุบันให้

## Slide 8: รู้จักกับ Constraint

Constraint คือกติกาที่ database ใช้ตรวจข้อมูลก่อนบันทึกลง table

ถ้าข้อมูลไม่ผ่านกติกา database จะปฏิเสธทันที แม้ frontend จะส่งข้อมูลเข้ามาแล้วก็ตาม

### ตัวอย่าง

```sql
title text not null check (char_length(title) >= 5)
```

### ความหมาย

- `not null`: ห้ามว่าง
- `check`: ต้องผ่านเงื่อนไข
- `char_length(title) >= 5`: title ต้องยาวอย่างน้อย 5 ตัวอักษร

Validation ควรมีหลายชั้น:

```text
frontend validation
server validation
database constraint
```

## Slide 9: รู้จักกับ Row Level Security

### Row Level Security คืออะไร

Row Level Security หรือ RLS คือระบบกำหนดสิทธิ์ระดับ row ใน database

- table เดียวกันอาจมีข้อมูลหลายแถว
- แต่ละ user หรือ role อาจเห็นข้อมูลไม่เท่ากัน
- RLS ใช้ policy บอกว่าใครอ่าน เขียน หรือแก้ไข row ไหนได้

ตัวอย่างเช่น user ควรเห็น issue ของตัวเอง ส่วน admin ควรเห็น issue ได้ทุกแถว

### สำหรับ Day 4

เรายังไม่มี login/auth จึงใช้ policy แบบ demo-only เพื่อให้เรียน CRUD ได้ก่อน แล้ว Day 5 ค่อยทำให้ปลอดภัยขึ้น

### Warning

Policy ต่อไปนี้เปิด public access สำหรับโปรเจกต์ฝึกเท่านั้น ห้ามใช้ production

## Slide 10: เปิด RLS และ Demo Policies

### File

```text
Supabase SQL Editor
```

### ตำแหน่งที่รัน

รันหลังจากสร้าง table `public.issues` สำเร็จแล้ว และก่อนเริ่มเรียกข้อมูลจาก Next.js

```sql
grant select, insert, update on public.issues to anon;

alter table public.issues enable row level security;

create policy "demo_select_issues"
  on public.issues
  for select
  to anon
  using (true);

create policy "demo_insert_issues"
  on public.issues
  for insert
  to anon
  with check (true);

create policy "demo_update_issues"
  on public.issues
  for update
  to anon
  using (true)
  with check (true);
```

### อ่านคำสั่งชุดนี้

- `grant select, insert, update on public.issues to anon;` ให้ role `anon` มีสิทธิ์อ่าน เพิ่ม และแก้ไขข้อมูลใน table ระดับ table ก่อน จากนั้น RLS policy จะเป็นตัวกำหนดว่าทำกับ row ไหนได้บ้าง
- `alter table public.issues enable row level security;` คือเปิด RLS ให้ table `issues`
- `create policy "..."` คือสร้างกติกาการเข้าถึงข้อมูล 1 ชุด
- `on public.issues` คือ policy นี้ใช้กับ table `issues`
- `for select` คือใช้กับการอ่านข้อมูล
- `for insert` คือใช้กับการเพิ่มข้อมูล
- `for update` คือใช้กับการแก้ไขข้อมูล
- `to anon` คือให้ role anonymous ใช้ policy นี้ได้ เวลา frontend เรียกผ่าน anon key
- `using (true)` คือเงื่อนไขสำหรับ row ที่อ่านหรือแก้ไขได้ ใน demo ใช้ `true` เพื่อเปิดให้ผ่านก่อน
- `with check (true)` คือเงื่อนไขสำหรับ row ที่เพิ่มหรือแก้ไขแล้วต้องผ่าน ใน demo ใช้ `true` เพื่อเปิดให้ผ่านก่อน

ชุดนี้เป็น policy สำหรับ demo เพื่อให้แอปลองอ่าน เพิ่ม และแก้ไขข้อมูลได้ก่อน พอขึ้น Day 5 เราจะเปลี่ยนจาก `true` เป็นเงื่อนไขที่ผูกกับ user/auth ให้ปลอดภัยขึ้น

### สรุปสั้น ๆ

`GRANT` กับ RLS ทำงานคนละชั้นกัน: `GRANT` ให้สิทธิ์เข้าถึง table ส่วน RLS policy คุมสิทธิ์ระดับ row ทั้งสองส่วนต้องตั้งค่าให้ครบ

### Day 5 จะเปลี่ยนอะไร

Day 5 จะเปลี่ยน policy ให้ผูกกับ authenticated user และ admin role โดยเน้น select/insert/update เป็น core

## Slide 13: Environment Variables

### ตำแหน่งที่วาง

สร้างไฟล์ `.env.local` ที่ project root ระดับเดียวกับ `package.json` ไม่ใช่ในโฟลเดอร์ย่อย

### Environment Variables คืออะไร

Environment variables คือค่าตั้งค่าที่โปรเจคต้องใช้ แต่เราไม่อยากเขียนฝังไว้ใน code โดยตรง

ใน slide นี้เราจะใส่ค่าที่ทำให้ Next.js รู้ว่าต้องเชื่อมต่อ Supabase project ไหน และต้องใช้ key ตัวไหนในการเรียก Data API

### ไปเอาค่าจาก Supabase ตรงไหน

1. เข้า Supabase Dashboard
2. เลือก project ที่สร้างไว้
3. กดปุ่ม `Connect` ด้านบนขวา
4. เลือก `Framework`
5. เลือก `Next.js`
6. เลือก `App Router`
7. ดูส่วน `.env.local` แล้วคัดลอก `NEXT_PUBLIC_SUPABASE_URL` และ `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` มาใส่ในไฟล์ `.env.local`

ถ้าหา key จากหน้า `Connect` ไม่เจอ สามารถกดไอคอนเฟือง `Settings` ด้านซ้าย แล้วเข้า `API Keys` เพื่อ copy `Publishable key` หรือ `anon public` ได้

`NEXT_PUBLIC_` หมายถึงค่าเหล่านี้ถูกใช้ฝั่ง browser ได้ จึงห้ามเอา secret key หรือ service role key มาใส่ตรงนี้

### ตัวอย่าง

```env
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=your-publishable-key
```

### หมายเหตุ

บาง project อาจยังใช้ชื่อ anon key:

```env
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key
```

ให้ใช้ key ที่ Supabase Dashboard หรือ Connect dialog แนะนำใน project ของผู้เรียน

## Slide 15: Update TypeScript Type

### File

```text
types/issue.ts
```

### ตำแหน่งที่แก้

ใช้ type ชุดนี้แทน type `IssueStatus` และ `Issue` เดิมในไฟล์ `types/issue.ts`

### ทำไมต้องแก้ type

ตอน Day 3 type `Issue` ใช้กับข้อมูลที่อยู่ใน React state เท่านั้น แต่ Day 4 ข้อมูลจะมาจาก Supabase table จริงแล้ว

ดังนั้น TypeScript type ต้องสะท้อน field ที่ database มี เช่น `createdAt`, `updatedAt`, `reporterEmail` และ `status` เพื่อให้ component, server action และ function ที่ query ข้อมูลใช้รูปแบบเดียวกัน

อีกจุดที่ต้องจำคือ database ใช้ชื่อแบบ snake_case เช่น `reporter_name` แต่ TypeScript ในแอปใช้ camelCase เช่น `reporterName` เดี๋ยวเราจะ map ค่าจาก database ให้เข้ากับ type นี้ตอนเขียน Supabase query

### Type

```ts
export type IssueStatus = "OPEN" | "IN_PROGRESS" | "DONE";

export type Issue = {
  id: string;
  reporterName: string;
  reporterEmail: string;
  title: string;
  description: string;
  status: IssueStatus;
  adminComment?: string;
  createdAt: string;
  updatedAt: string;
};
```

เพิ่ม `updatedAt` เพื่อให้ TypeScript ตรงกับ database มากขึ้น

### ถ้า `IssueForm.tsx` error

หลังเพิ่ม `updatedAt` แล้ว function ที่สร้าง issue จาก form ต้องใส่ field นี้ด้วย ไม่อย่างนั้น object ที่ return จะยังไม่ครบตาม type `Issue`

ให้ไปที่ `components/IssueForm.tsx` แล้วเติม `updatedAt` ใน object ที่ return จาก `createIssueFormInput`

```ts
function createIssueFormInput(input: NewIssueInput): Issue {
  const now = new Date().toISOString();

  return {
    id: crypto.randomUUID().slice(0, 8),
    reporterName: input.reporterName,
    reporterEmail: input.reporterEmail,
    title: input.title,
    description: input.description,
    status: "OPEN",
    createdAt: now,
    updatedAt: now,
  };
}
```

ใช้ตัวแปร `now` ร่วมกันเพื่อให้ `createdAt` และ `updatedAt` เป็นเวลาเดียวกันตอนสร้าง issue ใหม่

### ถ้า `data/issue.ts` error

ข้อมูลจำลองใน `data/issue.ts` ก็ประกาศเป็น `Issue[]` เช่นกัน ดังนั้นทุก object ต้องมี `updatedAt` ให้ครบตาม type ใหม่

ให้เติม `updatedAt` ต่อจาก `createdAt` ใน issue ทุกตัว โดยใช้ค่าเดียวกับ `createdAt` ได้ เพราะข้อมูลจำลองยังไม่เคยถูกแก้ไข

```ts
export const issues: Issue[] = [
  {
    id: "001",
    reporterName: "Anan",
    reporterEmail: "anan@example.com",
    title: "เข้าสู่ระบบไม่ได้",
    description: "กรอกรหัสผ่านแล้วระบบแจ้งว่าไม่ถูกต้อง",
    status: "OPEN",
    createdAt: "2026-07-14T09:00:00.000Z",
    updatedAt: "2026-07-14T09:00:00.000Z",
  },
];
```

ถ้าในไฟล์มีหลาย issue ต้องเติม `updatedAt` ให้ครบทุกตัว ไม่เช่นนั้น `npm run build` จะยังไม่ผ่าน

### ผลลัพธ์

Supabase project มี table `issues` พร้อมข้อมูล และ Next.js project มี env vars พร้อมเชื่อมต่อ

### SQL ที่ต้องรันครบใน Supabase

```sql
create type issue_status as enum ('OPEN', 'IN_PROGRESS', 'DONE');

create table issues (
  id uuid primary key default gen_random_uuid(),
  reporter_name text not null,
  reporter_email text not null,
  title text not null check (char_length(title) >= 5),
  description text not null,
  status issue_status not null default 'OPEN',
  admin_comment text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

alter table issues enable row level security;
```

### ไฟล์ใน Next.js ที่ต้องมีค่า env

```env
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=your-publishable-key
```
