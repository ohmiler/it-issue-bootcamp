## Slide 2: สร้าง `updateIssueStatus()` ใน Database Layer

`updateIssueStatus()` ทำหน้าที่เปลี่ยนสถานะของ issue หนึ่งรายการใน Supabase โดยรับ `id` ของ issue และสถานะใหม่ที่ต้องการบันทึก

### File

```text
lib/issues.ts
```

### ตำแหน่งที่วาง

วาง function `updateIssueStatus` ต่อจาก `createIssue` ใน `lib/issues.ts`

`IssueStatus` ถูก import อยู่แล้วจาก Hour 2–3 จึงไม่ต้องเพิ่ม import ซ้ำ ให้ตรวจว่าบรรทัดเดิมมี type นี้อยู่:

```ts
import type { Issue, IssueStatus, NewIssueInput } from "@/types/issue";
```

### Code

```ts
export async function updateIssueStatus(
  id: string,
  status: IssueStatus,
): Promise<void> {
  const supabase = createSupabaseServerClient();

  const { error } = await supabase
    .from("issues")
    .update({
      status,
      updated_at: new Date().toISOString(),
    })
    .eq("id", id);

  if (error) {
    throw new Error(`Failed to update issue: ${error.message}`);
  }
}
```

### อธิบายคำสั่ง

- `id: string` ระบุ issue ที่ต้องการแก้ไข
- `status: IssueStatus` จำกัดค่าที่ใช้ใน TypeScript ให้เป็น `OPEN`, `IN_PROGRESS` หรือ `DONE`
- `.from("issues")` เลือก table `issues`
- `.update({...})` กำหนด column และค่าใหม่
- `.eq("id", id)` อัปเดตเฉพาะ row ที่มี `id` ตรงกัน
- `if (error)` หยุดการทำงานเมื่อ Supabase update ไม่สำเร็จ

### ทำไมต้องแก้ `updated_at`

ค่า default `now()` ใน schema ทำงานตอน insert เท่านั้น โปรเจกต์นี้ยังไม่มี database trigger สำหรับแก้ `updated_at` อัตโนมัติ จึงกำหนดเวลาใหม่พร้อมกับการ update status

`.eq("id", id)` เป็นคำสั่งสำคัญ หาก update โดยไม่กรอง row อาจกระทบข้อมูลมากกว่าหนึ่งรายการ

## Slide 3: สร้าง Server Action สำหรับ Update Status

หลังสร้าง database function ใน Slide 2 เราจะเพิ่ม Server Action เพื่อรับคำสั่งจาก UI ตรวจข้อมูล และเรียก `updateIssueStatus()`

### File

```text
app/actions.ts
```

### ตำแหน่งที่วาง

แก้ import เดิมด้านบน `app/actions.ts` ให้รวม function และ type ใหม่ ไม่ต้องสร้าง import จาก path เดียวกันซ้ำ:

```ts
import { createIssue, updateIssueStatus } from "@/lib/issues";
import type { IssueStatus, NewIssueInput } from "@/types/issue";
```

จากนั้นวาง `allowedStatuses` และ `updateIssueStatusAction` ต่อจาก `createIssueAction`

### Code ใหม่ที่เพิ่ม

```ts
const allowedStatuses: IssueStatus[] = ["OPEN", "IN_PROGRESS", "DONE"];

export async function updateIssueStatusAction(
  formData: FormData,
): Promise<void> {
  const id = String(formData.get("id") ?? "").trim();
  const status = String(formData.get("status") ?? "").trim();

  if (!id || !allowedStatuses.includes(status as IssueStatus)) {
    throw new Error("Invalid update request");
  }

  await updateIssueStatus(id, status as IssueStatus);
  revalidatePath("/issues");
}
```

### ลำดับการทำงาน

1. อ่าน `id` และ `status` จาก `FormData`
2. ตรวจว่า `id` ไม่เป็นค่าว่าง
3. ตรวจว่า `status` อยู่ใน `allowedStatuses`
4. เรียก `updateIssueStatus()` เพื่อ update Supabase
5. เรียก `revalidatePath("/issues")` เพื่อให้หน้า list อ่านข้อมูลล่าสุด

### ทำไมต้องตรวจ `status` ซ้ำ

แม้ UI จะแสดงเฉพาะปุ่ม `OPEN`, `IN_PROGRESS` และ `DONE` ผู้ใช้ยังแก้ค่า hidden input หรือส่ง request เองได้ Server Action จึงต้องตรวจ allowlist ก่อนส่งค่าไป database ทุกครั้ง

Action นี้ไม่ใช้ `redirect()` เพราะผู้ใช้ยังอยู่หน้า `/issues` เดิม เราต้องการเพียงโหลดรายการใหม่หลัง update สำเร็จ

## Slide 5: ใครสามารถ Update Status ได้บ้าง

การย้าย update ไปไว้ใน Server Action ช่วยจัดระเบียบ logic แต่ไม่ได้ทำให้ระบบมี authorization โดยอัตโนมัติ

### สิทธิ์ของ Project ตอนนี้

- แอปยังไม่มีระบบ login
- Supabase client ใช้ Publishable key และทำงานด้วย role `anon`
- Hour 1 ให้สิทธิ์ `update` แก่ role `anon`
- demo RLS policy ใช้ `using (true)` และ `with check (true)`

```sql
create policy "demo_update_issues"
  on public.issues
  for update
  to anon
  using (true)
  with check (true);
```

ดังนั้นผู้ที่เข้าถึงแอปสามารถส่งคำสั่งเปลี่ยน status ของ issue ได้ทุก row นี่เป็น policy สำหรับการเรียน CRUD เท่านั้น

### ซ่อนปุ่มอย่างเดียวไม่พอ

ถึง UI จะแสดงปุ่มเฉพาะ admin ผู้ใช้ยังสามารถแก้ request หรือเรียก Server Action โดยตรงได้ การตรวจสิทธิ์จึงต้องเกิดฝั่ง server และ database ด้วย

### ระบบจริงควรเป็น

1. Authentication ตรวจว่า request มาจากผู้ใช้คนใด
2. Server Action ตรวจว่า user มี role ที่อนุญาตให้ update หรือไม่
3. RLS policy ตรวจซ้ำว่า role นั้นแก้ไข row นี้ได้จริง

ตัวอย่างสิทธิ์ที่ต้องการในระบบจริง:

- ผู้ใช้ทั่วไปสร้าง issue ได้
- ผู้ใช้ดู issue ของตัวเองได้
- admin หรือเจ้าหน้าที่ IT เท่านั้นที่เปลี่ยน status ได้
- ผู้ที่ไม่มีสิทธิ์ต้องถูกปฏิเสธ แม้จะส่ง request เอง

Day 4 ใช้ demo policy เพื่อโฟกัสที่ Update Flow ก่อน ส่วน Day 5 จะเพิ่ม authentication และ authorization ให้เหมาะกับระบบจริง

## Slide 6: ทำไมโปรเจกต์นี้ไม่ใช้ Hard Delete

### Hard Delete คืออะไร

Hard Delete คือการลบ row ออกจาก database ถาวร เช่น:

```sql
delete from public.issues
where id = 'issue-id';
```

เมื่อลบแล้ว query ปกติจะไม่สามารถอ่านข้อมูลนั้นกลับมาได้

### ทำไมไม่เหมาะกับระบบแจ้งปัญหา

- ต้องเก็บประวัติไว้ตรวจสอบย้อนหลัง
- เจ้าหน้าที่อาจต้องดูปัญหาเดิมเพื่อแก้ปัญหาที่เกิดซ้ำ
- ผู้ใช้อาจกดลบผิด
- ข้อมูลอาจเกี่ยวข้องกับรายงานการให้บริการหรือหลักฐานการทำงาน
- การลบต้องมีการตรวจสิทธิ์และการยืนยันที่รัดกุม

### ทางเลือกของโปรเจกต์นี้

```text
OPEN -> IN_PROGRESS -> DONE
```

เราใช้ status `DONE` เพื่อบอกว่า issue ถูกปิดงานแล้ว แต่ยังเก็บ row และประวัติไว้ใน database

การเปลี่ยน status เป็น `DONE` เป็นคำสั่ง Update ไม่ใช่ Delete เพียงแต่ทำหน้าที่แทนการนำ issue ออกจากงานที่กำลังดำเนินการ

### Close, Soft Delete และ Hard Delete ต่างกันอย่างไร

| วิธี | สิ่งที่เกิดกับข้อมูล |
|---|---|
| Close | เปลี่ยน status แต่ยังแสดงประวัติได้ |
| Soft Delete | ตั้งค่าเช่น `deleted_at` แล้วซ่อนจาก query ปกติ |
| Hard Delete | ลบ row ออกจาก database ถาวร |

ชั่วโมงนี้ implement เฉพาะ Close ด้วย status ส่วน Soft Delete และ Hard Delete เรียนในระดับแนวคิดเพื่อไม่ให้ scope ของโปรเจกต์ใหญ่เกินไป

## Slide 7: ทดสอบ Update Status

### ขั้นตอนทดสอบ

1. เปิดหน้า `/issues`
2. เลือก issue ที่ยังไม่ใช่สถานะ `DONE`
3. กดปุ่มสถานะใหม่ เช่น `IN_PROGRESS`
4. ตรวจว่า status บนหน้า list เปลี่ยน
5. refresh หน้าเว็บและตรวจว่าสถานะยังอยู่
6. เปิด Supabase Table Editor และตรวจค่า `status` กับ `updated_at`
7. เปลี่ยน status เป็น `DONE` เพื่อทดสอบการปิดงาน

### ผลลัพธ์ที่ควรได้

- ปุ่มของสถานะปัจจุบันถูก disable
- status ใหม่แสดงหลัง Server Action ทำงาน
- ค่าไม่ย้อนกลับหลัง refresh
- `updated_at` เปลี่ยนเป็นเวลาล่าสุด

### ถ้า Update ไม่สำเร็จ

- เห็น `permission denied`: ตรวจ `grant update` และ RLS update policy
- กดปุ่มแล้วไม่มีอะไรเกิดขึ้น: ตรวจ `action={updateIssueStatusAction}`
- เห็น `Invalid update request`: ตรวจ hidden input ชื่อ `id` และ `status`
- หน้าเปลี่ยนแต่ข้อมูลเก่า: ตรวจ `revalidatePath("/issues")`
- Supabase ไม่เปลี่ยน: ดู error ใน terminal และตรวจ `.eq("id", id)`

```text

```

3. เปิด production URL ที่ Vercel สร้างให้

Vercel จะติดตั้ง dependencies รัน production build และสร้าง URL สำหรับเปิดเว็บจริงให้เรา

ถ้าแก้ Environment Variables ภายหลัง ต้อง redeploy เพื่อให้ deployment ใหม่อ่านค่าล่าสุด

### ถ้า build fail

เปิด Build Logs แล้วอ่าน error แรกที่เกี่ยวข้อง จากนั้นแก้ในเครื่อง ทดสอบ `npm run build` แล้ว push เพื่อ deploy ใหม่

## Slide 11: ทดสอบ Production URL

### ตรวจหลัง deploy

- เปิด `/issues` และเห็นรายการจาก Supabase
- เปิดหน้ารายละเอียดของ issue
- สร้าง issue ใหม่จาก `/issues/new`
- เปลี่ยน status แล้ว refresh เพื่อตรวจว่าค่ายังอยู่
- เปิด Supabase Table Editor และตรวจว่าข้อมูลเปลี่ยนใน project เดียวกัน

### ถ้า local ใช้ได้ แต่ production ไม่ได้

มักเกี่ยวกับ:

- ตรวจชื่อและค่าของ env vars บน Vercel
- ตรวจว่า redeploy หลังแก้ env vars แล้ว
- อ่าน Runtime Logs ของ Vercel
- ตรวจ RLS policies และ grants ใน Supabase
- ตรวจว่า Vercel ใช้ repository และ branch ที่ถูกต้อง
- ตรวจว่า URL และ key มาจาก Supabase project เดียวกัน

หน้าแรกเปิดได้ยังไม่ถือว่าทดสอบเสร็จ ต้องลอง flow หลักตั้งแต่ Read, Create และ Update ด้วย

## Glossary

| คำ | ความหมาย |
|---|---|
| Update | การแก้ไขข้อมูล |
| Close issue | การปิดงานด้วย status เช่น `DONE` |
| Hard delete | ลบ record จริงจาก database |
| Soft delete | mark ว่าลบแล้ว แต่ยังเก็บ record |
| Build | การ compile project ก่อน deploy |
| Environment variables | ค่าตั้งค่าที่ไม่ควร hardcode ใน source code |
