## Slide 3: Server Action คืออะไร

Server Action คือ `async function` ที่ Next.js รันบน server ไม่ได้รันใน browser

ในโปรเจกต์นี้ เราจะใช้ Server Action รับข้อมูลจาก form ตรวจสอบความถูกต้อง และสั่งบันทึก issue ลง Supabase

### เมื่อผู้ใช้ submit form

1. browser ส่งข้อมูลจาก form ไปยัง server
2. Next.js เรียก Server Action ที่กำหนดไว้
3. Server Action อ่านและตรวจสอบข้อมูล
4. ข้อมูลที่ถูกต้องถูกบันทึกลง database
5. ระบบพาผู้ใช้ไปยังหน้าถัดไป

### ตัวอย่าง

```tsx
<form action={createIssueAction}>
  ...
</form>
```

`action={createIssueAction}` บอก Next.js ว่าเมื่อ form นี้ถูก submit ให้ส่งข้อมูลไปทำงานใน function `createIssueAction`

### ทำไมจึงใช้ Server Action

- เชื่อม form กับ logic ฝั่ง server ได้โดยตรง
- ติดต่อ database จากฝั่ง server ได้
- ไม่ต้องสร้าง API route แยกสำหรับการ submit form แบบพื้นฐาน

## Slide 5: อ่านข้อมูลจาก `FormData`

เมื่อผู้ใช้ submit form ฝั่ง server จะได้รับข้อมูลในรูปแบบ `FormData` เราจึงต้องอ่านค่าแต่ละช่องและแปลงให้อยู่ในรูปแบบ `NewIssueInput` ก่อน

### File

```text
app/actions.ts
```

### ตำแหน่งที่วาง

สร้างไฟล์ `app/actions.ts` ถ้ายังไม่มี แล้ววาง import และ helper `parseIssueInput` ไว้ด้านบนไฟล์ ก่อน server action หลัก

### Code

```ts
import type { NewIssueInput } from "@/types/issue";

function parseIssueInput(formData: FormData): NewIssueInput {
  return {
    reporterName: String(formData.get("reporterName") ?? "").trim(),
    reporterEmail: String(formData.get("reporterEmail") ?? "").trim(),
    title: String(formData.get("title") ?? "").trim(),
    description: String(formData.get("description") ?? "").trim(),
  };
}
```

### อธิบายคำสั่ง

- `formData.get("reporterName")` อ่านค่าจาก input ที่มี `name="reporterName"`
- `?? ""` ใช้ข้อความว่างเมื่อไม่พบค่านั้นใน form
- `String(...)` ทำให้ค่าที่อ่านได้เป็น string
- `.trim()` ตัดช่องว่างด้านหน้าและด้านหลังออก
- `NewIssueInput` กำหนดว่า function นี้ต้องคืนข้อมูลครบทั้ง 4 field

ชื่อที่ส่งให้ `formData.get(...)` ต้องตรงกับค่า `name` ของ input ใน `IssueForm`

ขั้นตอนนี้มีหน้าที่อ่านและจัดรูปแบบข้อมูลเท่านั้น ส่วนการตรวจว่าผู้ใช้กรอกข้อมูลถูกต้องหรือไม่จะทำใน Slide 6

## Slide 6: ตรวจสอบข้อมูลก่อนบันทึก

หลังจาก Slide 5 แปลง `FormData` เป็น `NewIssueInput` แล้ว ขั้นตอนต่อไปคือตรวจว่าข้อมูลครบและอยู่ในรูปแบบที่ยอมรับได้หรือไม่

เราต้องตรวจซ้ำที่ server เสมอ เพราะผู้ใช้สามารถข้าม validation ใน browser และส่งข้อมูลเข้ามาโดยตรงได้

### File

```text
app/actions.ts
```

### ตำแหน่งที่วาง

วาง function `validateIssueInput` ต่อจาก `parseIssueInput` และก่อน `createIssueAction`

### Code

```ts
function validateIssueInput(input: NewIssueInput): string[] {
  const errors: string[] = [];

  if (input.reporterName.length < 2) {
    errors.push("Reporter name is required");
  }

  if (!input.reporterEmail.includes("@")) {
    errors.push("Reporter email is invalid");
  }

  if (input.title.length < 5) {
    errors.push("Title must be at least 5 characters");
  }

  if (input.description.length < 10) {
    errors.push("Description must be at least 10 characters");
  }

  return errors;
}
```

### Function นี้ทำงานอย่างไร

- รับข้อมูล `NewIssueInput` ที่ได้จาก `parseIssueInput()`
- สร้าง `errors` เป็น array ว่างสำหรับเก็บข้อความผิดพลาด
- ตรวจชื่อผู้แจ้ง อีเมล หัวข้อ และรายละเอียดทีละเงื่อนไข
- ถ้าเงื่อนไขใดไม่ผ่าน จะเพิ่มข้อความลงใน `errors`
- คืนค่า `errors` หลังตรวจครบทุก field

### อ่านผลลัพธ์อย่างไร

- `errors.length === 0` หมายถึงข้อมูลผ่านทุกเงื่อนไข
- `errors.length > 0` หมายถึงยังมีข้อมูลที่ต้องแก้ และไม่ควรบันทึกลง database

Validation ใน Day 3 ช่วยให้ผู้ใช้เห็นคำเตือนได้รวดเร็ว ส่วน validation ฝั่ง server เป็นด่านสุดท้ายที่ต้องผ่านก่อนบันทึกข้อมูลจริง

## Slide 7: สร้าง `createIssue()` เพื่อบันทึกข้อมูล

`createIssue()` คือ function ในชั้น database ทำหน้าที่รับข้อมูลที่ผ่าน validation แล้ว และ insert issue ใหม่ลง table `issues` ใน Supabase

เราแยก function นี้ไว้ใน `lib/issues.ts` เพื่อไม่ให้ Server Action ต้องรู้รายละเอียดของชื่อ column ใน database

### File

```text
lib/issues.ts
```

### ตำแหน่งที่วาง

เพิ่ม `NewIssueInput` เข้าไปใน type import เดิมด้านบนไฟล์ แล้ววาง function `createIssue` ต่อจาก `getIssueById`

```ts
import type { Issue, IssueStatus, NewIssueInput } from "@/types/issue";
```

### Code

```ts
export async function createIssue(input: NewIssueInput) {
  const supabase = createSupabaseServerClient();

  const { error } = await supabase.from("issues").insert({
    reporter_name: input.reporterName,
    reporter_email: input.reporterEmail,
    title: input.title,
    description: input.description,
    status: "OPEN",
  });

  if (error) {
    throw new Error(`Failed to create issue: ${error.message}`);
  }
}
```

### Function นี้ทำงานอย่างไร

- รับ `input` ที่มี type เป็น `NewIssueInput`
- สร้าง Supabase client สำหรับติดต่อ database
- ใช้ `.from("issues")` เลือก table ที่ต้องการบันทึก
- ใช้ `.insert({...})` เพิ่ม row ใหม่ลงใน table
- กำหนด issue ใหม่ให้เริ่มต้นด้วยสถานะ `OPEN`
- ถ้า insert ไม่สำเร็จ จะ `throw new Error(...)` เพื่อหยุดการทำงาน

### ทำไมชื่อ field ไม่เหมือนกัน

ข้อมูลใน TypeScript ใช้รูปแบบ `camelCase` แต่ column ใน database ใช้ `snake_case` จึงต้อง map ชื่อก่อน insert:

- `reporterName` → `reporter_name`
- `reporterEmail` → `reporter_email`

เราอ่านเฉพาะ `error` จาก Supabase เพราะขั้นตอนนี้ต้องการรู้เพียงว่าบันทึกสำเร็จหรือไม่ ยังไม่จำเป็นต้องรับ row ที่สร้างกลับมา

## Slide 8: สร้าง Server Action สำหรับ Submit Form

`createIssueAction()` เป็น function ที่รับข้อมูลจาก form แล้วควบคุมขั้นตอนทั้งหมด ตั้งแต่อ่านข้อมูล ตรวจสอบ บันทึก และพาผู้ใช้กลับไปหน้ารายการ

### File

```text
app/actions.ts
```

### ตำแหน่งที่วาง

วาง `"use server";` เป็นบรรทัดแรกของ `app/actions.ts` ก่อน import ทุกบรรทัด จากนั้นวาง `createIssueAction` ต่อจาก helper `parseIssueInput` และ `validateIssueInput`

### Code

```ts
"use server";

import { revalidatePath } from "next/cache";
import { redirect } from "next/navigation";
import { createIssue } from "@/lib/issues";

export async function createIssueAction(formData: FormData) {
  const input = parseIssueInput(formData);
  const errors = validateIssueInput(input);

  if (errors.length > 0) {
    throw new Error(errors.join(", "));
  }

  await createIssue(input);

  revalidatePath("/issues");
  redirect("/issues");
}
```

### ลำดับการทำงาน

1. `parseIssueInput(formData)` อ่านข้อมูลจาก form และสร้าง `NewIssueInput`
2. `validateIssueInput(input)` ตรวจข้อมูลและคืนรายการ error
3. ถ้ามี error ให้หยุดการทำงานก่อนบันทึกลง database
4. `createIssue(input)` insert issue ใหม่ลง Supabase
5. `revalidatePath("/issues")` บอก Next.js ให้โหลดข้อมูลหน้ารายการใหม่
6. `redirect("/issues")` พาผู้ใช้กลับไปดูรายการ issue

### คำสั่งสำคัญ

- `"use server"` กำหนดว่า function ในไฟล์นี้ทำงานฝั่ง server
- `errors.join(", ")` รวมข้อความ error หลายรายการเป็นข้อความเดียว
- `await createIssue(input)` รอให้ insert เสร็จก่อนทำขั้นตอนถัดไป
- ต้องเรียก `revalidatePath()` ก่อน `redirect()` เพราะ `redirect()` จะจบการทำงานของ action ทันที

## Slide 9: เชื่อม `IssueForm` กับ Server Action

ใน Day 3 `IssueForm` ใช้ `handleSubmit` และ `onCreateIssue` เพื่อเพิ่มข้อมูลลง client state แต่ตอนนี้เราจะให้ form ส่งข้อมูลไปยัง `createIssueAction` โดยตรง

### File

```text
components/IssueForm.tsx
```

### ส่วนที่ลบจาก Day 3

- `import type { Issue } ...`
- import `useState`
- type `IssueFormProps`
- type `NewIssueInput` ที่ประกาศซ้ำใน component เพราะย้ายไปไว้ที่ `types/issue.ts` แล้ว
- function `getIssueInput()` เพราะ Server Action อ่าน `FormData` แทน
- function `validateIssueInput()` เพราะย้ายไปตรวจที่ server แล้ว
- function `createIssueFormInput()` เพราะ database จะสร้าง `id` และเวลาให้
- prop `onCreateIssue` จาก parameter ของ `IssueForm`
- state `errors` และ function `handleSubmit()`
- block `{errors.length > 0 && (...)}` ที่แสดง client error
- `onSubmit={handleSubmit}` บน `<form>`

ถ้าไฟล์มี `"use client"` ให้ลบออกด้วย แต่จากโค้ดปัจจุบันที่ส่งมาไม่มีบรรทัดนี้อยู่แล้ว

Logic สำหรับอ่านข้อมูลและ validation ย้ายไปอยู่ใน `app/actions.ts` แล้ว ส่วนการสร้าง issue ย้ายไปอยู่ใน `lib/issues.ts`

### ส่วนที่เก็บไว้

- JSX ของ input ทุกช่อง
- ค่า `name` ของ input
- label และข้อความบน form
- className และรูปแบบหน้าตาเดิม

### Code

```tsx
import { createIssueAction } from "@/app/actions";

export function IssueForm() {
  return (
    <form action={createIssueAction}>
      <fieldset className="mt-6 grid gap-5">
        <legend>ข้อมูลปัญหา</legend>

        {/* เก็บ input, label และ className เดิมทั้งหมด */}

        <button type="submit">ส่งข้อมูล</button>
      </fieldset>
    </form>
  );
}
```

ลบ `<section>` และ `<h2>แจ้งปัญหาใหม่</h2>` ที่ครอบ form ออกด้วย เพราะหน้า `/issues/new` จะเป็นผู้แสดงหัวข้อหลักและจัด layout ให้ ส่วน field และ className เดิมให้คงไว้ทั้งหมด

### เมื่อ Submit Form

1. browser รวบรวมค่าจาก input ที่มี `name`
2. form ส่ง `FormData` ไปยัง `createIssueAction`
3. Server Action ตรวจข้อมูลและบันทึกลง Supabase
4. เมื่อสำเร็จ ระบบ redirect กลับไปหน้า `/issues`

### ทำไมลบ `"use client"` ได้

หลังลบ `useState`, `handleSubmit` และ event handler แล้ว `IssueForm` ไม่มี code ที่ต้องทำงานใน browser จึงใช้เป็น Server Component ได้

ชั่วโมงนี้เราจะใช้ server validation ก่อน ส่วนการนำ error กลับมาแสดงใกล้ field ด้วย `useActionState` จะเป็นการต่อยอดภายหลัง

}

### 3. ลบ `IssueForm` ออกจาก JSX

```tsx
<IssueForm onCreateIssue={handleCreateIssue} />
```

ให้เก็บ `handleUpdateStatus`, `handleCloseIssue` และ `<IssueList />` ไว้ตามเดิม เพราะไม่ได้เกี่ยวกับ error นี้

แม้หน้า `/issues` จะไม่ได้เรียก `IssueBoard` แล้ว TypeScript ยังตรวจไฟล์ที่อยู่ใน project จึงต้องแก้ reference เก่านี้ให้ถูกต้อง

## Slide 12: ทดสอบ Create Flow จริง

หลังเชื่อม `IssueForm`, Server Action และ Supabase แล้ว ให้ทดสอบทั้งการบันทึกและการอ่านข้อมูลกลับมา

### ขั้นตอนทดสอบ

1. เปิดหน้า `/issues/new` และตรวจว่า form แสดงครบทุก field
2. กรอกข้อมูลให้ผ่านเงื่อนไข validation จาก Slide 6
3. กดปุ่ม `ส่งข้อมูล`
4. รอให้ระบบบันทึกข้อมูลและ redirect ไปหน้า `/issues`
5. ตรวจว่า issue ใหม่แสดงอยู่ในรายการ
6. refresh หน้า `/issues`
7. ตรวจว่า issue ยังอยู่ แสดงว่าข้อมูลถูกบันทึกใน Supabase ไม่ใช่ client state
8. เปิด Supabase Table Editor และตรวจว่า table `issues` มี row ใหม่

### ผลลัพธ์ที่ควรได้

- URL เปลี่ยนจาก `/issues/new` เป็น `/issues`
- issue ใหม่แสดงในหน้า list
- ข้อมูลไม่หายเมื่อ refresh
- row ใหม่มี `status` เป็น `OPEN`
- `id`, `created_at` และ `updated_at` มีค่าจาก database

### ถ้าสร้างไม่สำเร็จ

- เห็น validation error: ตรวจความยาวของชื่อ หัวข้อ รายละเอียด และรูปแบบอีเมล
- เห็น `permission denied`: ตรวจ `grant insert` และ RLS policy สำหรับ `insert`
- กดแล้วไม่ redirect: เปิด terminal และดู error จาก `createIssueAction()` หรือ `createIssue()`
- ข้อมูลบาง field ว่าง: ตรวจว่า `name` ของ input ตรงกับ `formData.get(...)`
- อ่าน env ไม่ได้: ตรวจ `.env.local` และ restart dev server
- redirect สำเร็จแต่ไม่เห็นข้อมูล: ตรวจ row ใน Supabase Table Editor และ query `getIssues()`

ควรตรวจ terminal ก่อน เพราะ error จาก Server Action และ Supabase จะแสดงรายละเอียดที่ช่วยหาสาเหตุได้เร็วที่สุด

### หมายเหตุเรื่อง Error UX

ถ้า validation ไม่ผ่าน ตอนนี้อาจเห็น Next.js error page เพราะ Server Action ใช้ `throw new Error(...)` ข้อมูลที่ผิดจะไม่ถูกบันทึกลง database แต่การแสดง error ใกล้ form ด้วย `useActionState` จะเป็นการต่อยอดภายหลัง

## Slide 13: ตรวจโค้ดสุดท้ายของ Create Flow

### `app/actions.ts`

ไฟล์นี้ต้องมี helper และ Server Action ครบในไฟล์เดียว โดย `"use server"` ต้องอยู่บรรทัดแรก:

```ts
"use server";

import { revalidatePath } from "next/cache";
import { redirect } from "next/navigation";
import { createIssue } from "@/lib/issues";
import type { NewIssueInput } from "@/types/issue";

function parseIssueInput(formData: FormData): NewIssueInput {
  return {
    reporterName: String(formData.get("reporterName") ?? "").trim(),
    reporterEmail: String(formData.get("reporterEmail") ?? "").trim(),
    title: String(formData.get("title") ?? "").trim(),
    description: String(formData.get("description") ?? "").trim(),
  };
}

function validateIssueInput(input: NewIssueInput): string[] {
  const errors: string[] = [];

  if (input.reporterName.length < 2) {
    errors.push("Reporter name is required");
  }

  if (!input.reporterEmail.includes("@")) {
    errors.push("Reporter email is invalid");
  }

  if (input.title.length < 5) {
    errors.push("Title must be at least 5 characters");
  }

  if (input.description.length < 10) {
    errors.push("Description must be at least 10 characters");
  }

  return errors;
}

export async function createIssueAction(formData: FormData) {
  const input = parseIssueInput(formData);
  const errors = validateIssueInput(input);

  if (errors.length > 0) {
    throw new Error(errors.join(", "));
  }

  await createIssue(input);
  revalidatePath("/issues");
  redirect("/issues");
}
```

ตรวจว่า `lib/issues.ts` มี function `createIssue(input)` จาก Slide 7 อยู่แล้ว เพราะ Server Action จะเรียก function นี้เพื่อ insert ลง Supabase

### `components/IssueForm.tsx`

ส่วนสำคัญที่เปลี่ยนคือ import Server Action และใช้กับ prop `action` ของ form ส่วน input เดิมให้เก็บไว้ครบ:

```tsx
import { createIssueAction } from "@/app/actions";

export function IssueForm() {
  return (
    <form action={createIssueAction}>
      {/* input และ fieldset เดิมทั้งหมด */}
      <button type="submit">ส่งข้อมูล</button>
    </form>
  );
}
```

ตัวอย่าง `IssueForm` ด้านบนแสดงเฉพาะจุดเชื่อม Create Flow ไม่ใช่ให้ลบ field จริงออกจากไฟล์

### ภาพรวม

```text
IssueForm
-> createIssueAction(formData)
-> validate ฝั่ง server
-> createIssue(input)
-> Supabase insert
-> revalidatePath("/issues")
-> redirect("/issues")
```

## Slide 14: ตรวจปัญหาที่พบบ่อย

### TypeScript หรือ Build Error

- `IssueBoard` ยังส่ง prop `onCreateIssue` ให้ `IssueForm`: ลบ create flow เก่าตาม Slide 10
- `IssueForm` ยังใช้ `useState` หรือ `handleSubmit`: ตรวจว่าได้ลบ client submit เดิมครบแล้ว
- `app/actions.ts` หา `parseIssueInput` หรือ `validateIssueInput` ไม่เจอ: ตรวจว่า helper ทั้งสองอยู่ก่อน `createIssueAction`
- ลืมวาง `"use server"` เป็นบรรทัดแรกของ `app/actions.ts`

### Submit แล้วเกิดปัญหา

- กดแล้วไม่เรียก Server Action: ตรวจว่า form ใช้ `action={createIssueAction}` ไม่ใช่ `onSubmit={handleSubmit}`
- ข้อมูลบาง field เป็นค่าว่าง: ตรวจ `name` ของ input ให้ตรงกับ `formData.get(...)`
- validation ไม่ทำงาน: ตรวจว่าเก็บผลลัพธ์ใน `errors` และเช็ก `errors.length` ก่อน insert

### Supabase หรือหน้า List ไม่อัปเดต

- เห็น `permission denied`: ตรวจ `grant insert` และ RLS policy สำหรับ `insert`
- insert แล้ว column บางตัวว่าง: ตรวจการ map `camelCase` เป็น `snake_case`
- redirect แล้วไม่เห็น issue ใหม่: ตรวจ `revalidatePath("/issues")` และ query `getIssues()`
- redirect ก่อนบันทึกเสร็จ: ต้อง `await createIssue(input)` ก่อน `redirect()`
- อ่าน env ไม่ได้: ตรวจ `.env.local` และ restart dev server

## Slide 15: สรุป Create Flow

### ก่อนเริ่มชั่วโมงนี้

`IssueForm` เพิ่มข้อมูลลง client state ข้อมูลจึงหายเมื่อ refresh หน้าเว็บ

### ตอนนี้ระบบทำงานอย่างไร

1. `IssueForm` ส่ง `FormData` ไปยัง `createIssueAction`
2. Server Action อ่านและ validate ข้อมูล
3. `createIssue()` insert row ใหม่ลง Supabase
4. `revalidatePath("/issues")` ทำให้หน้า list อ่านข้อมูลล่าสุด
5. `redirect("/issues")` พาผู้ใช้กลับไปดูรายการ

### ผลลัพธ์

- สร้าง issue ใหม่จากหน้า `/issues/new` ได้
- ข้อมูลถูกบันทึกใน Supabase
- issue ยังอยู่หลัง refresh
- database เป็น source of truth แทน client state

### ต่อไป

ชั่วโมงถัดไปเราจะเพิ่มการ update status ใน Supabase เรียนรู้แนวคิด soft delete และเตรียม deploy ไป Vercel
