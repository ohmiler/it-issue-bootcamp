## Slide 2: CRUD จากมุม Frontend

### Create

ผู้ใช้กรอก form แล้ว issue ใหม่ปรากฏใน list

### Read

ผู้ใช้เห็นรายการ issue และดูรายละเอียดได้

### Update

Admin เปลี่ยน status เช่น:

```text
OPEN -> IN_PROGRESS -> DONE
```

### Delete ในระบบนี้

Hard delete คือการลบ record ออกจากระบบจริง ๆ เช่น ลบ issue ออกจาก database ทำให้รายการนั้นหายไปจาก list และย้อนดูประวัติได้ยาก

ไม่ทำ hard delete เป็น core แต่ใช้แนวคิด:

```text
Close issue -> เปลี่ยน status เป็น DONE
Soft delete -> ต่อเติมภายหลังถ้าต้องการ
```

ก่อนเชื่อม database ควรเข้าใจว่า UI ต้องรองรับ action อะไรบ้าง แต่ไม่จำเป็นต้องทำ hard delete ในระบบแจ้งปัญหาขนาดเล็ก

## Slide 3: เพิ่ม Handler สำหรับ Update Status

### File

```text
components/IssueBoard.tsx
```

### Import ที่ต้องใช้

ถ้าไฟล์ `IssueBoard.tsx` ยัง import แค่ `Issue` ให้เพิ่ม `IssueStatus` เข้ามาด้วย

```tsx
import type { Issue, IssueStatus } from "@/types/issue";
```

### ตำแหน่งที่วาง

วาง function `handleUpdateStatus` ภายใน `IssueBoard` ต่อจาก `useState` และก่อน `return`

### Code

```tsx
function handleUpdateStatus(id: string, status: IssueStatus) {
  setItems((currentItems) =>
    currentItems.map((issue) =>
      issue.id === id
        ? {
            ...issue,
            status,
          }
        : issue
    )
  );
}
```

ใน React ควร update state แบบ immutable ไม่แก้ object เดิมตรง ๆ

## Slide 6: Close Issue Handler

### File

```text

```

### อธิบาย

`handleCloseIssue` คือ function ที่ใช้ตอนผู้ใช้กดปุ่มปิด issue โดยเราไม่ลบข้อมูลออกจาก list แต่เปลี่ยน status ของ issue นั้นเป็น `DONE` แทน

ในชั่วโมงนี้เรามี `handleUpdateStatus` อยู่แล้ว จึงให้ `handleCloseIssue` เรียก function เดิมซ้ำ แล้วส่ง status เป็น `"DONE"` เพื่อให้โค้ดอ่านง่ายขึ้นว่า action นี้คือ “ปิดงาน”

### ตำแหน่งที่วาง

วาง function `handleCloseIssue` ต่อจาก `handleUpdateStatus` และก่อน `return`

### Code

```tsx
function handleCloseIssue(id: string) {
  handleUpdateStatus(id, "DONE");
}
```

สำหรับระบบแจ้งปัญหา การ close issue ด้วย status มักเหมาะกว่า hard delete เพราะยังเก็บประวัติไว้ได้

## Slide 9: จาก Mock State ไป Database

### ตอนนี้

### สิ่งที่จะเปลี่ยน

## Slide 10: รู้จักกับ Server Actions

### Server Action คืออะไร

Server Action คือ async function ที่รันฝั่ง server และถูกเรียกเพื่อทำงานแบบ mutation เช่น create, update หรือ delete ข้อมูล

ในเอกสาร Next.js เวอร์ชันนี้ คำกว้างคือ Server Function ส่วน Server Action คือ Server Function ที่เอามาใช้กับ action ของ form หรือ mutation

### ทำไมโปรเจกต์นี้จะได้ใช้

ระบบแจ้งปัญหาของเรามี action ที่ต้องเปลี่ยนข้อมูลจริง:

- สร้าง issue ใหม่จาก `IssueForm`
- update status
- close issue

งานเหล่านี้ไม่ควรจบแค่ `useState` ใน browser เพราะ refresh แล้วหาย และ client ไม่ควรเป็นคนตัดสินใจเรื่อง validation หรือ permission เอง

### Preview ของ Day 4

ยังไม่ต้องใส่ใน project ตอนนี้ ใช้เป็นภาพล่วงหน้าว่า `IssueForm` จะเปลี่ยนจาก `onSubmit` ฝั่ง client ไปเป็น `<form action={...}>`

```tsx
async function createIssue(formData: FormData) {
  "use server";

  const title = formData.get("title");
  // validate
  // save to database
}

export function IssueForm() {
  return <form action={createIssue}>...</form>;
}
```

Server Action ทำให้ form ส่งข้อมูลไป logic ฝั่ง server ได้โดยตรง รับ `FormData` อัตโนมัติ แล้วค่อย validate, บันทึกลง database และ refresh/revalidate ข้อมูลที่แสดงบนหน้า

## Slide 14: โค้ดสุดท้ายของ Mock Update/Close

### `components/IssueBoard.tsx`

หลังประกอบครบแล้ว `IssueBoard` จะถือ state และ handler สำหรับ update กับ close:

```tsx
import type { Issue, IssueStatus } from "@/types/issue";
```

```tsx
const [items, setItems] = useState<Issue[]>(initialIssues);

function handleUpdateStatus(id: string, status: IssueStatus) {
  setItems((currentItems) =>
    currentItems.map((issue) =>
      issue.id === id ? { ...issue, status } : issue
    )
  );
}

function handleCloseIssue(id: string) {
  handleUpdateStatus(id, "DONE");
}

```

### JSX ใน `IssueBoard`

จุดที่คุมความกว้างของเนื้อหาหลักคือ `max-w-*` บน `<main>` เดิมใช้ `max-w-5xl` แต่พอ table มี column จัดการเพิ่มเข้ามา ควรขยับเป็น `max-w-6xl` เพื่อให้ข้อมูลไม่ถูกบีบเกินไป

```tsx
return (
  <main className="mx-auto grid grid-cols-1 max-w-6xl gap-6 px-6 py-8">
    <IssueForm onCreateIssue={handleCreateIssue} />

    <IssueList
      issues={items}
      onUpdateStatus={handleUpdateStatus}
      onCloseIssue={handleCloseIssue}
    />
  </main>
);
```

### ถ้าปรับแล้วความกว้างยังไม่เปลี่ยน

ให้เปิด `app/globals.css` แล้วเช็ก CSS เดิมจาก Day 1 ถ้ายังมี selector แบบนี้อยู่ มันจะ override ความกว้างของ Tailwind บน `<main>`

```css
main {
  max-width: 760px;
  margin: 0 auto;
  padding: 24px;
}
```

ให้ลบ `max-width` และ `padding` ของ `main` ออกจาก global CSS เพราะตอนนี้เราคุม layout ด้วย class บน `<main>` แล้ว เช่น `max-w-6xl`, `px-6`, `py-8`

อย่าเพิ่งลบ global CSS จนเหลือแค่นี้ ถ้า component บางตัวยังพึ่ง CSS เดิมอยู่ เช่น `IssueForm` ยังใช้ `<section>` ที่ไม่มี class หรือยังใช้ `.form-row`

ก่อน cleanup ให้เช็กว่า style สำคัญถูกย้ายเข้า component แล้ว เช่น:

```tsx
<section className="rounded-lg border border-slate-200 bg-white p-6">
```

```tsx
<footer className="px-6 py-8 text-center text-sm text-slate-500">
```

ถ้ายังไม่เปลี่ยน `.form-row` เป็น Tailwind ให้เก็บเฉพาะ `.form-row` ใน `app/globals.css` ไว้ชั่วคราว ไม่อย่างนั้นช่องชื่อและอีเมลจะกลับมาเรียงแนวตั้ง

```css
.form-row {
  display: grid;
  grid-template-columns: repeat(2, minmax(0, 1fr));
  gap: 16px;
}
```

หลังจากย้าย component ที่พึ่ง CSS เดิมครบแล้ว ค่อย cleanup `app/globals.css` ให้เหลือประมาณนี้ เพื่อไม่ให้ selector กว้าง ๆ ชนกับ Tailwind

```css
@import "tailwindcss";

body {
  margin: 0;
  background: #f6f7fb;
  color: #111827;
  line-height: 1.5;
}
```

ส่วน selector กว้าง ๆ อย่าง `main`, `section`, `button`, `table`, `input`, `textarea` ควรลบออกเมื่อ component นั้นมี Tailwind class คุมเองแล้ว

## Slide 15: Common Mistakes

### ข้อผิดพลาดที่พบบ่อย

- แก้ state ตรง ๆ เช่น `issue.status = "DONE"`
- ลืมใช้ `.map()` เพื่อสร้าง array ใหม่
- ส่ง handler ลง component ไม่ครบ
- คิดว่าข้อมูลใน `useState` คือข้อมูลถาวร
- ลืมว่า update/close ต้องตรวจสิทธิ์ฝั่ง server ในระบบจริง
- ปรับ `max-w-6xl` แล้วไม่เห็นผล เพราะ `app/globals.css` ยังมี `main { max-width: ... }` จาก CSS เดิม
- ลบ global CSS เร็วเกินไป ทั้งที่ `IssueForm`, `.form-row` หรือ `footer` ยังไม่ได้ย้ายมาเป็น Tailwind

## Glossary

| คำ | ความหมาย |
|---|---|
| Mock state | ข้อมูลจำลองที่เก็บใน browser memory |
| Immutable update | การสร้างข้อมูลใหม่แทนการแก้ object เดิมตรง ๆ |
| Close issue | การปิดงานด้วย status เช่น `DONE` |
| Soft delete | การซ่อนหรือ mark ว่าลบแล้ว โดยไม่ลบ record จริง |
| Server Action | function ฝั่ง server ที่ form หรือ component เรียกได้ |
