# Login, Logout, and Protected Pages

## Day 5 - ชั่วโมงที่ 2: ทำให้ผู้ใช้ Login และป้องกันหน้าที่ต้องมีสิทธิ์

### เป้าหมายของชั่วโมงนี้

หลังเรียนจบ ผู้เรียนจะสามารถ:

1. Login ด้วย email และ password ผ่าน Supabase Auth
2. อ่านผู้ใช้ปัจจุบันจากฝั่ง server
3. ป้องกันทั้ง page และ Server Action ที่ต้อง login
4. แสดง navigation ตามสถานะ login
5. Logout และล้าง session ได้

### ไฟล์ที่ใช้

```text
app/login/page.tsx                 สร้างใหม่
components/LoginForm.tsx           สร้างใหม่
app/actions.ts                     เพิ่ม login, logout และ auth check
lib/auth.ts                        สร้างใหม่
app/issues/new/page.tsx            แก้บางส่วน
components/AppNav.tsx              สร้างใหม่
app/layout.tsx                     แก้บางส่วน
```

## Slide 1: จาก SSR Client สู่ Login จริง

Hour 1 เตรียม Supabase client ฝั่ง Browser และ Server รวมถึง `proxy.ts` สำหรับอัปเดต session cookie แล้ว

Hour นี้เราจะต่อ flow ให้ครบ:

```text
กรอก email/password
        ↓
Supabase ตรวจบัญชีผู้ใช้
        ↓
เก็บ session ไว้ใน cookie
        ↓
Server รู้ว่าใครกำลังใช้งาน
        ↓
อนุญาตให้เปิดหน้าและเรียก action ที่ป้องกันไว้
```

เราจะโฟกัส email/password, session และการป้องกันระบบก่อน ส่วน register และ OAuth ยังไม่อยู่ในชั่วโมงนี้

## Slide 2: เตรียม Account สำหรับทดลอง

ไปที่ **Supabase Dashboard → Authentication → Users** แล้วสร้างผู้ใช้ไว้ล่วงหน้า:

```text
user@example.com
admin@example.com
```

กำหนด password สำหรับทดลอง และตรวจให้ email เป็น confirmed

### ทำไมยังไม่ทำหน้า Register

- ต้องจัดการ email confirmation เพิ่ม
- อาจเสียเวลากับ inbox หรือ spam
- เป้าหมายตอนนี้คือ login, session และ protected page

ทั้งสองบัญชียังเป็นเพียงผู้ใช้ที่ login ได้ ส่วน role `USER` และ `ADMIN` จะสร้างในชั่วโมงถัดไป

## Slide 3: Login Flow ทำงานอย่างไร

เราแบ่งหน้าที่เป็น 3 ส่วน:

1. `LoginForm` รับ email และ password
2. `signInAction` ส่งข้อมูลไปตรวจที่ Supabase Auth
3. Supabase SSR client เขียน session ลง cookie แล้ว redirect ไป `/issues`

```text
LoginForm ──FormData──> signInAction
                              │
                              ▼
                  signInWithPassword()
                              │
                    สำเร็จ ──┴── ไม่สำเร็จ
                      │              │
                 /issues      /login?error=...
```

password จะถูกส่งเข้า Server Action โดยตรง เราไม่ต้องเก็บ password ไว้ใน state หรือ database ของแอป

## Slide 4: สร้างหน้า Login

### สร้างไฟล์ใหม่ทั้งไฟล์

```text
app/login/page.tsx
```

```tsx
import { LoginForm } from "@/components/LoginForm";

type LoginPageProps = {
  searchParams: Promise<{ error?: string }>;
};

export default async function LoginPage({ searchParams }: LoginPageProps) {
  const { error } = await searchParams;

  return (
    <main className="mx-auto max-w-md px-6 py-12">
      <section className="rounded-lg border border-slate-200 bg-white p-6">
        <h1 className="text-2xl font-bold text-slate-950">เข้าสู่ระบบ</h1>
```

## Slide 5: สร้าง Login Form

### สร้างไฟล์ใหม่ทั้งไฟล์

```text
components/LoginForm.tsx
```

```tsx
"use client";

import { useEffect } from "react";
import { signInAction } from "@/app/actions";

type LoginFormProps = {
  showError: boolean;
};

export function LoginForm({ showError }: LoginFormProps) {
  useEffect(() => {
    if (showError) {
      // ลบ error ออกจาก URL เพื่อให้ข้อความหายเมื่อ Refresh ครั้งถัดไป
      window.history.replaceState(null, "", "/login");
    }
  }, [showError]);

  return (
    <>
      {showError && (
        <p className="mt-4 text-sm font-medium text-red-700">
          อีเมลหรือรหัสผ่านไม่ถูกต้อง
        </p>
      )}

      <form action={signInAction} className="mt-6 grid gap-4">
        <label className="grid gap-2 text-sm font-semibold text-slate-800">
          Email
          <input
            name="email"
            type="email"
            autoComplete="email"
            required
            className="rounded-md border border-slate-300 px-3 py-2"
          />
        </label>

        <label className="grid gap-2 text-sm font-semibold text-slate-800">
          Password
          <input
            name="password"
            type="password"
            autoComplete="current-password"
            required
            className="rounded-md border border-slate-300 px-3 py-2"
          />
        </label>

        <button className="rounded-md bg-teal-700 px-4 py-3 text-sm font-bold text-white">
          เข้าสู่ระบบ
        </button>
      </form>
    </>
  );
}
```

- `"use client"` ทำให้ component ใช้ `useEffect()` และ Browser History API ได้
- `replaceState()` เปลี่ยน URL เป็น `/login` โดยไม่ reload หน้า ข้อความจึงยังอยู่ในรอบปัจจุบัน
- เมื่อผู้ใช้ Refresh อีกครั้ง URL ไม่มี `error` แล้ว `showError` จึงเป็น `false` และข้อความจะหายไป
- `action={signInAction}` ยังส่งค่าจาก input เป็น `FormData` ไปยัง Server Action เหมือนเดิม

## Slide 6: เพิ่ม Login Action

### แก้บางส่วนในไฟล์เดิม

```text
app/actions.ts
```

เพิ่ม import ที่ยังไม่มี แล้ววาง function ต่อจาก actions เดิม:

เมื่อผู้ใช้กดปุ่ม Login ค่า `email` และ `password` จากฟอร์มจะถูกส่งเข้ามาใน `formData` ส่วน action นี้ทำงานฝั่ง server จึงใช้ Supabase Server Client เพื่อตรวจบัญชีและบันทึก session cookie ได้

```ts
import { createClient } from "@/lib/supabase/server";

export async function signInAction(formData: FormData) {
  const email = String(formData.get("email") ?? "").trim();
  const password = String(formData.get("password") ?? "");

  if (!email || !password) {
    redirect("/login?error=invalid-credentials");
  }

  const supabase = await createClient();
  const { error } = await supabase.auth.signInWithPassword({ email, password });

  if (error) {
    console.error("Login failed:", error.message);
    redirect("/login?error=invalid-credentials");
  }

  // รีเฟรชข้อมูลใน layout ให้แสดงสถานะ login ล่าสุด
  revalidatePath("/", "layout");
  redirect("/issues");
}
```

### โค้ดทำงานทีละขั้น

1. อ่านและตรวจว่า `email` กับ `password` ไม่เป็นค่าว่าง
2. `signInWithPassword()` ส่งข้อมูลไปให้ Supabase Auth ตรวจสอบ
3. ถ้าไม่สำเร็จ ให้ log สาเหตุจริงใน terminal แล้วกลับหน้า Login ด้วยข้อความทั่วไป
4. รีเฟรช layout ให้เห็นสถานะใหม่ แล้วพาผู้ใช้ไปหน้า `/issues`

ถ้าสำเร็จ Supabase จะเก็บ session ใน cookie ไฟล์นี้มี `redirect` และ `revalidatePath` จากบทก่อนแล้ว จึงไม่ต้อง import ซ้ำ

หน้าเว็บแสดงข้อความกลาง ๆ เพื่อไม่เปิดเผยข้อมูลบัญชี ส่วน `console.error()` แสดงสาเหตุจริงเฉพาะใน terminal ฝั่ง server และต้องไม่ log password

## Slide 7: รองรับผู้ใช้ที่ Login แล้ว

### ทบทวนจาก Day 4

- ผู้ใช้ที่ยังไม่ Login จะส่ง request ด้วย role `anon`
- demo policy ใช้เงื่อนไข `true` จึงทำให้ทุกคนสามารถ `select`, `insert` และ `update` ได้
- เมื่อ Login สำเร็จ Supabase จะเปลี่ยน request ไปใช้ role `authenticated`

ปัญหาคือ policy เดิมระบุไว้เฉพาะ `anon` คนที่ Login แล้วจึงอาจเข้าใช้งานได้ แต่ query ตาราง `issues` ไม่ผ่าน

ตอนนี้เราจะเพิ่ม `authenticated` ให้ใช้งานได้ชั่วคราวควบคู่กับ `anon` ก่อน แล้วค่อยบังคับให้เฉพาะผู้ที่ Login และมีสิทธิ์ถูกต้องใน Hour 3

รันใน **Supabase SQL Editor**:

```sql
grant select, insert, update on public.issues to authenticated;

drop policy if exists "demo_select_issues" on public.issues;
drop policy if exists "demo_insert_issues" on public.issues;
drop policy if exists "demo_update_issues" on public.issues;

create policy "demo_select_issues"
on public.issues for select
to anon, authenticated
using (true);

create policy "demo_insert_issues"
on public.issues for insert
to anon, authenticated
with check (true);

create policy "demo_update_issues"
on public.issues for update
to anon, authenticated
using (true)
with check (true);
```

นี่ยังเป็น **demo policy ที่เปิดกว้าง** ชั่วโมงถัดไปเราจะลบและแทนด้วย policy ที่ตรวจ user และ role จริง

## Slide 8: รู้ว่าใครกำลัง Login

หลังจาก Login แล้ว session ถูกเก็บไว้ใน cookie แต่แต่ละ page หรือ action ยังต้องถาม Supabase ว่า session นี้เป็นของผู้ใช้คนใด

เราจึงสร้าง helper กลาง 2 แบบ เพื่อไม่ต้องเขียนโค้ดตรวจ user ซ้ำทุกไฟล์:

- `getCurrentUser()` ใช้เมื่อ **อยากรู้** ว่ามีใคร Login อยู่หรือไม่
- `requireUser()` ใช้เมื่อ feature นั้น **บังคับว่าต้อง Login**

### สร้างไฟล์ใหม่ทั้งไฟล์

```text
lib/auth.ts
```

```ts
import { redirect } from "next/navigation";
import { createClient } from "@/lib/supabase/server";

export async function getCurrentUser() {
  const supabase = await createClient();
  const { data, error } = await supabase.auth.getUser();

  if (error || !data.user) {
    return null;
  }

  return data.user;
}

export async function requireUser() {
  const user = await getCurrentUser();

  if (!user) {
    redirect("/login");
  }

  return user;
}
```

### ลำดับการทำงาน

1. `createClient()` สร้าง Supabase client ที่อ่าน session cookie ของ request นี้
2. `getUser()` ส่ง token ไปตรวจสอบกับ Supabase Auth server
3. ถ้า token ถูกต้อง จะได้ข้อมูล `user`; ถ้าไม่ถูกต้องหรือไม่มี session จะคืน `null`
4. `requireUser()` เรียก `getCurrentUser()` อีกที และ redirect ไป `/login` เมื่อไม่พบ user

### เลือกใช้ให้เหมาะกับงาน

```text
Navigation          → getCurrentUser()  เพราะผู้ที่ยังไม่ Login ก็เปิดดูได้
Protected Page      → requireUser()     เพราะต้อง Login ก่อนเท่านั้น
Protected Action    → requireUser()     เพราะต้องป้องกันฝั่ง server
```

## Slide 9: ป้องกันหน้า `/issues/new`

หน้า `/issues/new` ใช้สร้างข้อมูลใหม่ จึงไม่ควรให้ผู้ที่ยังไม่ Login เปิดฟอร์มได้ แม้จะพิมพ์ URL เข้ามาโดยตรงก็ตาม

### แก้บางส่วนในไฟล์เดิม

```text
app/issues/new/page.tsx
```

ไม่ต้องลบ `<section>`, heading หรือข้อความเดิม ให้เพิ่มเพียง import และ `await requireUser()` ก่อน `return`

### โค้ดหลังแก้ทั้งไฟล์

```tsx
import { IssueForm } from "@/components/IssueForm";
import { requireUser } from "@/lib/auth";

export default async function NewIssuePage() {
  await requireUser();

  return (
    <main className="mx-auto max-w-3xl px-6 py-8">
      <section className="rounded-lg border border-slate-200 bg-white p-6">
        <h1 className="text-2xl font-bold text-slate-950">แจ้งปัญหาใหม่</h1>
        <p className="mt-2 text-sm text-slate-600">
          กรอกข้อมูลปัญหาที่ต้องการให้ฝ่าย IT ตรวจสอบ
        </p>
        <IssueForm />
      </section>
    </main>
  );
}
```

### เมื่อเปิดหน้านี้จะเกิดอะไรขึ้น

```text
มี user     → requireUser() คืนข้อมูล user → render IssueForm
ไม่มี user  → requireUser() redirect        → ไปหน้า /login
```

- `NewIssuePage` ต้องเป็น `async` เพราะเรารอผลจาก `requireUser()` ด้วย `await`
- วาง auth check ก่อน `return` เพื่อไม่ให้ React render หรือส่งฟอร์มกลับไปก่อนตรวจเสร็จ
- การตรวจใน Server Component ป้องกันการเข้าหน้าผ่าน URL โดยตรง ไม่ใช่แค่ซ่อนลิงก์ `New Issue`

ตอนนี้เราป้องกัน “หน้า” ได้แล้ว แต่ยังต้องป้องกัน Server Action ที่รับข้อมูลจากฟอร์มด้วยในสไลด์ถัดไป

## Slide 10: ป้องกัน Server Action ด้วย

Slide 9 ป้องกันไม่ให้ผู้ที่ยังไม่ Login เห็นฟอร์ม แต่ไม่ได้แปลว่า `createIssueAction` ปลอดภัยโดยอัตโนมัติ

Server Action เป็น function ฝั่ง server ที่รับ request จากภายนอก ผู้ใช้อาจส่ง request มาเองโดยไม่เปิดหน้า `/issues/new` ก่อนก็ได้ ดังนั้น action ต้องตรวจ user ของตัวเองทุกครั้ง

### แก้บางส่วนใน `app/actions.ts`

เพิ่ม import แล้ววาง auth check เป็นบรรทัดแรกของ `createIssueAction`:

```ts
import { requireUser } from "@/lib/auth";

export async function createIssueAction(formData: FormData) {
  await requireUser();

  // เก็บโค้ด parse, validate และ createIssue เดิมไว้ด้านล่าง
}
```

วาง `await requireUser()` ไว้บรรทัดแรก ก่อนอ่าน `formData` หรือเขียน database:

```text
เรียก createIssueAction
          ↓
     requireUser()
       ↙       ↘
ไม่มี user      มี user
redirect        parse → validate → insert
```

```text
ป้องกัน Page    → คนที่ยังไม่ login มองไม่เห็น form
ป้องกัน Action  → คนที่ยังไม่ login สร้างข้อมูลไม่ได้
```

ตอนนี้เราตรวจเฉพาะ Authentication ว่า Login แล้วหรือยัง ส่วนการตรวจ Authorization ว่าใครมีสิทธิ์สร้างหรือแก้ข้อมูลใด จะเพิ่มใน Hour 3

## Slide 11: เพิ่ม Logout Action

Login เป็นการสร้าง session ส่วน Logout คือการยกเลิก session เพื่อไม่ให้ request ถัดไปถูกมองว่าเป็นผู้ใช้คนเดิม

ปุ่ม Logout ที่เราจะสร้างใน Slide 12 จะอยู่ใน `<form>` และเรียก `signOutAction` โดยตรง จึงไม่ต้องสร้าง API route เพิ่ม

### เพิ่มต่อจาก `signInAction` ในไฟล์เดิม

```text
app/actions.ts
```

```ts
export async function signOutAction() {
  const supabase = await createClient();
  const { error } = await supabase.auth.signOut();

  if (error) {
    redirect("/issues?error=logout-failed");
  }

  // ให้ layout อ่าน session ใหม่และเปลี่ยนเป็นปุ่ม Login
  revalidatePath("/", "layout");
  redirect("/login");
}
```

### โค้ดทำงานอย่างไร

1. สร้าง Supabase Server Client ที่อ่าน session cookie ปัจจุบัน
2. `signOut()` ยกเลิก session และลบข้อมูลการ Login ออกจาก cookie
3. ถ้าไม่สำเร็จ ให้กลับ `/issues` พร้อม `error` ใน URL
4. ถ้าสำเร็จ ให้รีเฟรช layout แล้ว redirect ไป `/login`

หลัง Logout หากเปิด `/issues/new` อีกครั้ง `requireUser()` จะไม่พบ user และส่งกลับหน้า Login ทันที

## Slide 12: Navigation ตามสถานะ Login

Navigation ควรเปลี่ยนตามสถานะของผู้ใช้ เพื่อไม่แสดงคำสั่งที่ใช้งานไม่ได้:

```text
ยังไม่ Login  → แสดง Login
Login แล้ว    → แสดง New Issue, email และ Logout
```

เราจะใช้ `getCurrentUser()` เพราะ navigation เปิดได้ทั้งสองกรณี ต่างจาก protected page ที่ต้องใช้ `requireUser()`

### สร้างไฟล์ใหม่ทั้งไฟล์

```text
components/AppNav.tsx
```

```tsx
import Link from "next/link";
import { signOutAction } from "@/app/actions";
import { getCurrentUser } from "@/lib/auth";

export async function AppNav() {
  const user = await getCurrentUser();

  return (
    <nav
      className="flex items-center justify-between border-b px-6 py-3"
    >
      <Link href="/issues" className="font-bold">
        Issue Tracker
      </Link>

      <div className="flex items-center gap-4 text-sm">
        {user ? (
          <>
            <Link href="/issues/new" className="font-semibold">
              New Issue
            </Link>
            <span>{user.email}</span>
            <form action={signOutAction}>
              <button className="font-semibold text-teal-700">
                Logout
              </button>
            </form>
          </>
        ) : (
          <Link href="/login" className="font-semibold text-teal-700">
            Login
          </Link>
        )}
      </div>
    </nav>
  );
}
```

### จุดสำคัญในโค้ด

- `AppNav` เป็น Server Component จึงเรียก `getCurrentUser()` และอ่าน session ฝั่ง server ได้โดยตรง
- ถ้า `user` มีค่า เงื่อนไข `user ? ... : ...` จะแสดงข้อมูลผู้ใช้และปุ่ม Logout
- ถ้า `user` เป็น `null` จะแสดงลิงก์ Login แทน
- `<form action={signOutAction}>` ทำให้ปุ่ม Logout เรียก Server Action โดยไม่ต้องเขียน event handler

Navigation ช่วยให้ผู้ใช้เห็นคำสั่งที่เหมาะกับสถานะ แต่ไม่ใช่ security boundary—page และ action ยังต้องตรวจสิทธิ์ของตัวเองเสมอ

## Slide 13: ใส่ Navigation ใน Layout

`RootLayout` เป็นโครงสร้างที่ครอบทุก route ของแอป ถ้าวาง `AppNav` ที่นี่ หน้า Home, Login และ Issues จะใช้ navigation ชุดเดียวกันโดยไม่ต้อง import ซ้ำทุกหน้า

### แก้บางส่วนใน `app/layout.tsx`

ไม่ต้องแทนทั้งไฟล์ ให้เก็บ `globals.css`, fonts, metadata และ class เดิมทั้งหมดไว้ แล้วเพิ่มเพียงสองจุด

### จุดที่ 1: เพิ่ม import ด้านบน

```tsx
import { AppNav } from "@/components/AppNav";
```

### จุดที่ 2: วางก่อน `{children}`

```tsx
<body>
  <AppNav />
  {children}
</body>
```

- `<AppNav />` คือ navigation ที่แสดงเหมือนกันทุกหน้า
- `{children}` คือเนื้อหาของ route ปัจจุบัน เช่น LoginPage หรือ IssuesPage
- หลัง Login หรือ Logout เราเรียก `revalidatePath("/", "layout")` เพื่อให้ `AppNav` อ่าน session และแสดงเมนูใหม่

อย่าเรียก `requireUser()` ใน root layout เพราะหน้า `/login` ก็ใช้ layout นี้ด้วย อาจทำให้เกิดการ redirect กลับหน้า Login ซ้ำไปมา

## Slide 14: ทดสอบ Authentication Flow ให้ครบ

อย่าทดสอบเฉพาะกรณี Login สำเร็จ ให้ลองทั้งสถานะก่อน Login, หลัง Login และหลัง Logout

### ทำตามลำดับ

1. เปิด `/issues/new` ขณะที่ยังไม่ Login → ต้องถูกส่งไป `/login`
2. กรอก password ผิด → ต้องเห็นข้อความผิดพลาด
3. ดู URL ต้องกลับเป็น `/login` แล้วกด Refresh → ข้อความต้องหาย
4. Login ด้วย account ที่ถูกต้อง → ต้องไป `/issues`
5. Navigation ต้องแสดง `New Issue`, email และ `Logout`
6. เปิด `/issues/new` และสร้าง issue → ต้องบันทึกข้อมูลได้
7. กด `Logout` → ต้องกลับไป `/login` และ navigation แสดงปุ่ม `Login`
8. เปิด `/issues/new` อีกครั้ง → ต้องถูกส่งกลับ `/login`

### ถ้าผลลัพธ์ไม่ตรง

```text
Login ไม่ผ่าน
→ ตรวจ Auth Users, password, confirmed email และข้อความใน terminal

Login ผ่าน แต่ query issues ไม่ได้
→ ตรวจว่าได้รัน SQL สำหรับ authenticated ใน Slide 7

Login/Logout แล้ว navigation ไม่เปลี่ยน
→ ตรวจ revalidatePath("/", "layout") ใน action
```

## Slide 15: จบ Authentication ก่อนเริ่ม Authorization

```text
Login → Session Cookie → Current User → Protected Page/Action → Logout
```

### ตอนนี้ระบบทำอะไรได้แล้ว

- Login และ Logout ผ่าน Server Action
- อ่าน user ปัจจุบันจาก session cookie ด้วย `getCurrentUser()`
- บังคับให้ Login ด้วย `requireUser()`
- ป้องกันทั้งหน้า `/issues/new` และ `createIssueAction`
- แสดง navigation ตามสถานะของผู้ใช้

### แต่ยังไม่จบเรื่อง Security

ตอนนี้เราตอบได้แล้วว่า **“ผู้ใช้คนนี้คือใคร”** ซึ่งคือ Authentication แต่ยังไม่ได้ตอบว่า **“ผู้ใช้คนนี้ทำอะไรกับข้อมูลใดได้บ้าง”** ซึ่งคือ Authorization

demo policy ยังอนุญาตกว้าง และ issue ยังไม่ผูกกับเจ้าของข้อมูล จึงยังไม่ใช่สิทธิ์สำหรับ production

### ชั่วโมงถัดไป

เราจะเพิ่ม `profiles`, role `USER` / `ADMIN`, เจ้าของ issue และ RLS เพื่อให้ database ตรวจสิทธิ์จริง ไม่ได้พึ่งเพียงการซ่อนปุ่มใน UI
