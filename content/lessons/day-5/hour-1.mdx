# Auth Flow and Supabase SSR

## Slide 2: รู้จัก Authentication และ Authorization

สองคำนี้มักใช้คู่กัน แต่ทำหน้าที่คนละขั้น

### Authentication: ผู้ใช้คือใคร

เป็นขั้นตอนตรวจสอบตัวตน เช่น ตรวจ email และ password ตอน login

```text
Login สำเร็จ → ระบบรู้ว่าเป็น user คนไหน
```

### Authorization: ผู้ใช้ทำอะไรได้

หลังจากรู้แล้วว่าเป็นใคร ระบบจะตรวจต่อว่า user คนนั้นมีสิทธิ์ทำอะไร

```text
USER สร้าง issue ได้
ADMIN เปลี่ยน status ได้
```

สรุปคือ Authentication ตรวจตัวตนก่อน ส่วน Authorization ตรวจสิทธิ์หลังจากนั้น

## Slide 3: Auth Flow ทำงานอย่างไร

### 1. ผู้ใช้ Login

หน้า Login ส่ง email และ password ไปให้ Supabase Auth ตรวจสอบ

### 2. ระบบสร้าง Session

ถ้าข้อมูลถูกต้อง Supabase จะสร้าง session และเก็บ token ไว้ใน cookie

### 3. ผู้ใช้เปิดหน้าหรือสั่งทำงาน

Browser ส่ง cookie มากับ request ทำให้ Server Component และ Server Action รู้ว่าเป็น user คนไหน

### 4. RLS ตรวจสิทธิ์ก่อนเข้าถึงข้อมูล

Supabase ใช้ตัวตนของ user ตรวจ RLS Policy ก่อนอนุญาตให้อ่านหรือแก้ database

Auth จึงไม่ได้จบแค่หน้า Login แต่ทำงานต่อไปจนถึงชั้น database

## Slide 4: รู้จัก Session, Token และ Cookie

### Session: ระบบจำว่า Login แล้ว

หลังจาก login สำเร็จ ผู้ใช้ไม่ควรต้องกรอก password ใหม่ทุกครั้งที่เปิดหน้า ระบบจึงสร้าง session เพื่อจำสถานะการ login

### Token: ข้อมูลสำหรับยืนยันตัวตน

- `access token` ใช้ยืนยันตัวตนกับ request ปัจจุบัน
- `refresh token` ใช้ขอ access token ใหม่เมื่อชุดเดิมหมดอายุ

### Cookie: ส่ง Token ไปกับ Request

Browser ส่ง cookie มาให้ Next.js อัตโนมัติ ทำให้ Server Component และ Server Action รู้ว่า request นี้มาจาก user คนไหน

สรุปคือ session บอกสถานะการ login, token ใช้ยืนยันตัวตน และ cookie ช่วยส่ง token ระหว่าง browser กับ server

## Slide 5: รู้จัก `@supabase/ssr`

`@supabase/ssr` คือ package ที่ช่วยให้ Supabase Auth ทำงานร่วมกับเว็บที่ render ทั้งฝั่ง browser และ server เช่น Next.js App Router

### ทำไมเราต้องใช้

ถ้าเก็บ session ไว้เฉพาะใน `localStorage` ฝั่ง browser จะรู้ว่าใคร login แต่ Server Component และ Server Action จะอ่าน session นั้นไม่ได้

`@supabase/ssr` จึงย้ายการจัดการ session มาใช้ cookies ทำให้ browser และ server อ่านข้อมูลการ login ชุดเดียวกันได้

### Package นี้ช่วยอะไร

- สร้าง browser client สำหรับ Client Component
- สร้าง server client สำหรับ Server Component และ Server Action
- ทำงานร่วมกับ Proxy เพื่อดูแล token ให้พร้อมใช้งาน

ผลลัพธ์คือไม่ว่า code จะทำงานฝั่งไหน ระบบก็รู้ว่า request มาจาก user คนใด

## Slide 9: สร้าง Server Client ที่อ่าน Cookie ได้

### แทนทั้งไฟล์: `lib/supabase/server.ts`

Server client เดิมจาก Day 4 เชื่อม database ได้ แต่ยังไม่รู้ว่า user คนไหนส่ง request มา จึงต้องแทนด้วย code นี้:

```ts
import { createServerClient } from "@supabase/ssr";
import { cookies } from "next/headers";

export async function createClient() {
  const cookieStore = await cookies();

  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY!,
    {
      cookies: {
        getAll() {
          return cookieStore.getAll();
        },
        setAll(cookiesToSet, _headers) {
          try {
            cookiesToSet.forEach(({ name, value, options }) =>
              cookieStore.set(name, value, options)
            );
          } catch {
            // Server Component เขียน cookie ไม่ได้ในบางจังหวะ
            // Proxy จะเป็นผู้ refresh และบันทึก cookie แทน
          }
        },
      },
    }
  );
}
```

### ส่วนสำคัญ

- `cookies()` อ่าน cookies ของ request ปัจจุบัน
- `getAll()` ส่ง cookies ที่มีอยู่ให้ Supabase อ่าน session
- `setAll()` บันทึก token ชุดใหม่เมื่อ session ถูก refresh
- `catch` ปล่อยผ่านกรณีที่ Server Component เขียน cookie ไม่ได้ เพราะ Proxy ดูแลงานนี้แทน

Server Component, Server Action และ Route Handler จะใช้ client จากไฟล์นี้

หลังแทนไฟล์ อาจเห็น error ว่า `createSupabaseServerClient` ไม่มีอยู่แล้ว เพราะ `lib/issues.ts` ยัง import ชื่อเดิมอยู่ เป็น error ที่คาดไว้และจะแก้ทันทีใน Slide 10

## Slide 10: แก้ Error หลังเปลี่ยน Server Client

### แก้บางส่วน: `lib/issues.ts`

Error เกิดจาก `server.ts` เปลี่ยนชื่อ export แล้ว แต่ `lib/issues.ts` ยังเรียกชื่อเดิม

### ก่อน

```ts
import { createSupabaseServerClient } from "./supabase/server";

const supabase = createSupabaseServerClient();
```

### หลัง

```ts
import { createClient } from "@/lib/supabase/server";

const supabase = await createClient();
```

### ถ้า `.from()` ขึ้น Error

ข้อความ `Property 'from' does not exist on type 'Promise'` หมายความว่าลืมใส่ `await` หน้า `createClient()`

แก้ทั้ง `getIssues()`, `getIssueById()`, `createIssue()` และ `updateIssueStatus()` ทุก function ต้องมี `await createClient()` แล้วตรวจว่า error หายก่อนทำขั้นต่อไป

## Slide 11: รู้จัก Next.js Proxy

Proxy คือ function ที่ทำงานหลังจาก request เข้ามา แต่ก่อน Next.js เปิดหน้าหรือเรียก Server Action

### ทำไม Auth ต้องใช้ Proxy

Server Component อ่าน cookie ได้ แต่เขียน cookie ใหม่ไม่ได้ในบางจังหวะ Proxy จึงเข้ามาช่วยดูแล session ก่อน request ไปถึงหน้าเว็บ

### Proxy ทำอะไรบ้าง

1. อ่าน token จาก cookies ของ request
2. ตรวจ token และ refresh เมื่อจำเป็น
3. ส่ง cookies ชุดใหม่ให้ทั้ง server และ browser
4. ปล่อยให้ request ทำงานต่อ

Proxy ดูแล session เท่านั้น ส่วนการตรวจว่าผู้ใช้มีสิทธิ์เปิดหน้าหรือทำ action หรือไม่ ต้องตรวจในหน้าและ Server Action อีกครั้ง

## Slide 12: สร้าง Helper สำหรับดูแล Session

### ไฟล์ใหม่: `lib/supabase/proxy.ts`

ไฟล์นี้รวม logic ที่ Proxy จะเรียกใช้กับทุก request โดย function `updateSession()` รับ request เข้ามา ตรวจ session แล้วส่ง response กลับไป

เริ่มด้วยโครงไฟล์นี้ก่อน:

```ts
import { createServerClient } from "@supabase/ssr";
import { NextResponse, type NextRequest } from "next/server";

export async function updateSession(request: NextRequest) {
  let response = NextResponse.next({ request });

  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY!,
    {
      cookies: {
        getAll() {
          return request.cookies.getAll();
        },
        setAll(cookiesToSet, headers) {
          // Slide 13 จะเติมส่วนส่ง cookies ชุดใหม่
        },
      },
    }
  );

  await supabase.auth.getClaims();

  return response;
}
```

ตอนนี้ helper อ่าน cookies ด้วย `getAll()` และตรวจ token ด้วย `getClaims()` แล้ว แต่ยังไม่ได้เขียน cookies ชุดใหม่กลับไป ขั้นนั้นจะเติมใน Slide 13

## Slide 13: ส่ง Cookies ชุดใหม่ให้ Server และ Browser

### แก้บางส่วน: `lib/supabase/proxy.ts`

เมื่อ Supabase refresh token จะส่งรายการ `cookiesToSet` เข้ามา เราต้องอัปเดต cookie ทั้งใน request และ response เพื่อให้ server กับ browser ใช้ session ชุดเดียวกัน

แทน comment `// Slide 13 จะเติมส่วนส่ง cookies ชุดใหม่` ด้วย code นี้:

```ts
cookiesToSet.forEach(({ name, value }) =>
  request.cookies.set(name, value)
);

response = NextResponse.next({ request });

cookiesToSet.forEach(({ name, value, options }) =>
  response.cookies.set(name, value, options)
);

Object.entries(headers).forEach(([key, value]) =>
  response.headers.set(key, value)
);
```

### อ่านตามลำดับ

1. `request.cookies` ทำให้ code ฝั่ง server เห็น token ชุดใหม่ทันที
2. `NextResponse.next({ request })` สร้าง response จาก request ที่อัปเดตแล้ว
3. `response.cookies` ส่ง token ชุดใหม่กลับไปเก็บใน browser
4. `response.headers` ป้องกัน response ที่มี session ถูก cache แล้วส่งผิด user

## Slide 14: เปิดใช้งาน Helper ด้วย Root Proxy

ไฟล์ `lib/supabase/proxy.ts` เป็นเพียง helper และ Next.js จะยังไม่เรียกใช้เอง เราต้องสร้าง `proxy.ts` เป็น entry point ที่ project root

### ไฟล์ใหม่: `proxy.ts`

สร้างไฟล์ระดับเดียวกับ `app/` และ `package.json` แล้วใช้ code นี้เป็นเนื้อหาทั้งไฟล์:

```ts
import { type NextRequest } from "next/server";
import { updateSession } from "@/lib/supabase/proxy";

export async function proxy(request: NextRequest) {
  return await updateSession(request);
}

export const config = {
  matcher: [
    "/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)",
  ],
};
```

- function `proxy()` ส่ง request ต่อให้ `updateSession()` ที่สร้างไว้
- `matcher` ให้ Proxy ทำงานกับหน้าเว็บ แต่ข้ามไฟล์ static และรูปภาพที่ไม่ใช้ session

ไม่ต้องจำ regular expression ใน `matcher` ให้คัดลอกตามตัวอย่างและเข้าใจหน้าที่โดยรวมก่อน

## Slide 16: Error ที่อาจเจอและวิธีแก้

### อ่านข้อความ Error ก่อนแก้

- `Cannot find module '@supabase/ssr'` → รัน `npm install @supabase/ssr`
- `Export createSupabaseServerClient doesn't exist` → เปลี่ยน import เป็น `createClient`
- `Property 'from' does not exist on type 'Promise'` → เติม `await createClient()`
- `Supabase environment variables are missing` → ตรวจ `.env.local` แล้ว restart dev server
- `Failed to fetch issues` → ตรวจ env และ demo policy จาก Day 4

แก้ error แรกให้หายก่อน แล้วจึงตรวจ error ถัดไป เพราะ error หลังอาจเกิดตามมาจากสาเหตุเดียวกัน
